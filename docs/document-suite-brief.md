# 五份产品文档基础信息

> 项目：AI 知识付费直播复盘
> 项目性质：个人独立项目
> 使用方式：把本文直接提供给新任务，用于继续撰写需求调研、竞品分析、PRD、评测文档和评测集。

## 一、项目基础事实

### 一句话介绍

主播提交直播链接或直播录屏后，系统生成带时间戳的逐字稿，结合违禁词规则、AI 知识付费话术框架、RAG 和大模型语义分析，定位风险原话和直播节奏问题，并给出下一场可以直接使用的整改话术。

### 目标用户

1. AI 课程、训练营、陪跑类知识付费主播。
2. 负责主播复盘、合规和转化优化的直播运营。
3. 管理多名主播、需要查看历史报告的内部管理员。

### 核心使用场景

- 主播下播后，希望快速复盘一场长直播。
- 运营需要定位哪一分钟、哪句话存在风险。
- 团队需要判断干货、课程承接、案例和成交环节是否完整。
- 主播需要把高风险表达改成仍有转化力、但更安全的话术。
- 管理员需要按主播沉淀和回看历史报告。

### 当前判断的用户痛点

1. 人工反复拖动长录屏，定位问题非常耗时。
2. 普通违禁词工具只能匹配词，无法理解收益暗示、案例夸大等上下文风险。
3. 合规分析和直播转化节奏通常分开进行。
4. 很多报告只说哪里有问题，不给主播可直接使用的改法。
5. 复盘结果分散，无法按主播长期沉淀。

注意：以上是基于场景和产品经验形成的待验证假设，目前不能写成已经完成访谈并证实的研究结论。

## 二、当前产品方案

### 用户流程

```text
内部账号登录
-> 选择直播链接或上传录屏
-> 直播链接录制完成后手动开始分析
-> 阿里 OSS 文件中转
-> 阿里 ASR 生成带时间戳逐字稿
-> 选择内置框架或输入自定义框架
-> 违禁词规则 + RAG + DeepSeek + LangGraph 分析
-> 生成总览、风险、节奏、改稿和追问报告
-> 报告保存到当前主播历史记录
-> 可同步飞书
```

### 产品原则

- ASR 时间戳判断真实时长，文字量只用于判断信息密度。
- 每个风险必须尽量包含原话、时间点、风险原因和替换话术。
- 本地规则识别确定性风险，大模型负责上下文语义判断。
- DeepSeek 或 Embedding 失败时保留规则和关键词兜底。
- 主分析使用固定五节点 LangGraph 工作流；节奏、整改和报告追问使用受控 ReAct，不做自由多 Agent 对话。
- 短录屏尚未进入后续时间阶段时，不强行判断后续环节缺失。

### 报告结构

1. 总览：安全分、关键结论、优先处理项和行动清单。
2. 风险整改：原话、时间点、类型、等级、依据和替换说法。
3. 节奏时间轴：阶段时间窗、实际覆盖、逐字稿和风险标签。
4. 数据复盘：当前为明确标注的模拟数据。
5. 复盘改稿：主播下一场可直接使用的整段话术。
6. 追问报告：围绕本场内容继续问，并返回参考时间点。

## 三、需求调研报告需要写什么

### 核心研究问题

1. 主播现在如何复盘，一场直播要花多长时间？
2. 主播最在意违规、节奏、转化还是改稿？
3. 哪些问题必须定位到时间点才有价值？
4. 主播是否愿意上传完整录屏，能接受多久出报告？
5. 主播会不会使用替换话术，判断“可播”的标准是什么？
6. 历史报告是主播自己看，还是运营和负责人看？
7. 用户是否愿意为单场、按月或团队账号付费？

### 建议访谈对象

- 5-8 名 AI 知识付费主播。
- 3-5 名直播运营或场控。
- 2-3 名负责多个主播的管理者。

### 调研输出

- 用户画像和角色差异。
- 当前复盘流程图。
- 痛点出现频率、严重程度和现有解决方式。
- 用户原话证据。
- 需求优先级。
- 已验证结论、被否定假设和仍待验证问题。

不要虚构访谈人数、用户原话和调研结论。

## 四、竞品分析报告需要写什么

### 建议分析四类产品

1. 平台违规词或内容合规检测工具。
2. 直播数据分析平台。
3. ASR、会议转写和音视频总结工具。
4. AI 直播话术生成或改写工具。

### 对比维度

- 支持直播链接还是只支持文件。
- 是否提供时间戳逐字稿。
- 只查关键词还是支持上下文语义风险。
- 是否支持行业话术框架。
- 是否分析直播节奏。
- 是否提供可直接使用的替换话术。
- 是否支持追问报告。
- 是否按主播保存历史记录。
- 是否接入真实直播数据。
- 输入限制、生成耗时、价格和服务对象。

### 本项目差异化假设

本项目不是单一违禁词工具，也不是单一数据看板，而是把“时间点逐字稿、风险判断、节奏框架和可播改稿”放进同一份播后报告中。

竞品功能、价格和最新状态会变化，正式撰写时必须重新联网核实并标注来源日期。

## 五、项目 PRD 需要写什么

### 产品目标

- 缩短人工复盘耗时。
- 提高高风险原话的定位效率。
- 让主播获得可直接执行的整改建议。
- 按主播沉淀可回看的历史资产。

### 非目标

- 不替代平台官方审核或法律判断。
- 不做实时直播拦截。
- 当前不做通用全行业平台。
- 不做视频画面、评论区和商品审核。
- 不保证外部录制工具的平台级稳定性。

### MVP 核心模块

1. 内部账号与主播权限。
2. 直播链接录制与录屏上传。
3. 带时间戳 ASR。
4. 内置框架和单场自定义框架。
5. 违禁词与语义风险分析。
6. 节奏时间轴。
7. 整改建议和复盘改稿。
8. 历史报告。
9. 报告追问。
10. 飞书同步。

### 建议北极星指标

每场报告中被主播确认采用的整改建议数。

当前尚未实现“确认采用”埋点，可以先观察：

- 真实录屏分析成功率。
- 报告生成耗时。
- 历史报告回访率。
- 复盘改稿查看率和复制率。
- 报告追问发起率。

## 六、评测文档需要写什么

### 评测对象

1. ASR 文字和时间点。
2. 违禁词和语义风险识别。
3. 框架阶段和节奏判断。
4. RAG 召回依据。
5. 替换话术质量。
6. 报告追问的事实一致性。
7. 工程稳定性、耗时和成本。

### 基线方案

- 纯关键词规则。
- DeepSeek，不使用 RAG。
- DeepSeek + RAG。
- 规则 + RAG + DeepSeek 完整方案。

### 指标

- ASR：字错误率、风险关键词召回率、时间戳误差。
- 风险识别：Precision、Recall、F1，按风险类型统计。
- 风险等级：与人工标签一致率。
- 框架阶段：阶段分类和缺失判断准确率。
- RAG：Recall@K、依据相关性和错误召回率。
- 改写：安全性、忠实度、可播性、具体性，各 1-5 分。
- 追问：事实一致性、时间点引用准确率、无依据拒答率。
- 工程：结构化输出成功率、任务成功率、平均耗时和单场成本。

参考目标值只能写成目标，不能冒充实测结果：

- 确定性违禁词召回率不低于 95%。
- 语义风险 Precision 不低于 80%。
- 结构化输出成功率不低于 99%。
- 改写人工评分均值不低于 4/5。

## 七、评测集需要怎么建

### 建议第一版规模

先做 40-50 条人工标注 Case，保证每条都能解释，而不是追求大而空。

### Case 分布

| Case 类型 | 建议数量 |
| --- | ---: |
| 明确违禁词和结果保证 | 8 |
| 隐性收益承诺和案例夸大 | 8 |
| AI 工具能力夸大和焦虑成交 | 6 |
| 合规表达、否定表达和引用表达 | 8 |
| 框架阶段与短录屏判断 | 6 |
| 跨段上下文风险 | 4 |
| ASR 噪声、口音和专业词 | 4 |
| 报告追问与无依据拒答 | 4 |

### 每条 Case 的字段

```text
case_id
audio_or_text_source
transcript
start_seconds
end_seconds
expected_risk_type
expected_risk_level
expected_framework_stage
expected_evidence
expected_rewrite
expected_rag_document_ids
reviewer
review_status
dataset_version
```

### 最适合讲解的四个代表 Case

1. **确定性高风险**：“保证你一个月用 AI 变现。”期望识别为结果承诺并弱化确定收益。
2. **反例控制**：“我不能保证每个人都能赚钱。”期望不因出现“保证、赚钱”而误报。
3. **案例边界**：“三十岁宝妈边带孩子边做副业，很快拿到结果。”需要结合背景、投入和不可复制性判断。
4. **短录屏节奏**：10 分钟片段尚未进入课程承接阶段，期望返回“暂不判断”，而不是直接判定缺失。

### 评测集原则

- 训练、调试和最终测试样本要分开。
- 每次修改 Prompt、规则或知识库后使用固定回归集。
- 记录模型、Prompt、知识库和评测集版本。
- 评测集必须包含合规反例，否则只能测召回，无法测误报。
- 不使用未经授权的真实直播内容公开展示。

## 八、五份文档的逻辑关系

```text
需求调研：证明问题是不是真的
-> 竞品分析：证明现有方案哪里不足
-> PRD：定义我们具体做什么
-> 评测文档：定义 AI 做到什么程度才算好
-> 评测集：用固定 Case 证明结果，而不是只靠主观演示
```
